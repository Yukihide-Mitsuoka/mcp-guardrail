## 1. 背景および動機(問題意識の定義)

### 1.1 MCP(Model Context Protocol)の本質認識

- AIモデル(脳)、MCPクライアント(UI / Cursor・Claude等)、MCPサーバー(手足 / API・データ連携)の3層構造。
- AIモデルに外部の文脈(Context)を安全かつ標準化された方法で提供するための共通ルール(Protocol)。
- MCP仕様は「接続(Authentication)」までは規定するが、「何のToolをどこまで実行できるか(Authorization)」はほぼサーバー実装に委ねられている。

### 1.2 公式および既存OSSのMCPサーバーの脆弱性(ガバガバ問題)

- 現状のOSSはPoC(概念実証)や個人ローカル利用が前提。
- 環境変数にフルアクセス(管理者権限)のトークン/PWを渡すケースがほとんどで、AIに全権限が移譲されている。
- 読み取り専用(`SELECT`のみ)の制御すらなく、プロンプトインジェクションでデータ破壊(`DROP`等)されるリスクがある。
- マルチテナント(複数ユーザーでの権限切り替え)を想定したコードが皆無。

### 1.3 AIエージェントの特性による経済的・セキュリティ的リスク

- AIはタスク完了を最優先するため、手綱を握らないと悪気なく全期間・数百万件のデータを取得しようとする。
- APIの上限超過、クラウド(BigQuery等)のデータスキャン量暴走による高額請求(破産リスク)。
- **(v2追記)** 単発の巨大リクエストだけでなく、**リトライループによる小さいリクエストの大量発行**でも累積コストは暴走する。1リクエスト単位の統制だけでは不十分。

### 1.4 市場の空白地帯(3レベルモデル)

| レベル | 機能 | 現状(2026) |
|---|---|---|
| Level 1 | MCPサーバーの集約・認証・ルーティング | 多数ある(Microsoft MCP Gateway, Docker MCP Gateway, ContextForge等) |
| Level 2 | Tool単位の認可(RBAC)・監査 | 出始めている(Envoy AI Gateway, Kuadrant(CEL), AgentGateway, Permit等) |
| Level 3 | **引数レベル・業務ルール・組織横断ガバナンス** | **ほぼ存在しない ← MCP GuardRailのターゲット** |

## 2. ビジネス要件・戦略

### 2.1 プロダクト・ブランディング

- **名称:** 『MCP GuardRail(MCPガードレール)』
- **選定理由:** 難解なインフラ用語(Gateway等)を避け、非エンジニア(経営層・情シス)に対して「AIの暴走、コスト破産、情報漏洩という崖から転落するのを物理的に止める安全装置」という安心感と提供価値を直感的に伝えるため。
- **ポジショニング(MCP Gatewayとの差別化):**
  - *MCP Gateway:* 道路(あらゆるAPIとLLMを高速・安全に「繋ぐ/開通する」汎用ルーティング。インフラ層向け)。
  - *MCP GuardRail:* 安全装置(データ・コスト特化の検閲、事故らせない防御。経営・予算管理者向け)。
- 思想的にはGatewayよりも **AWS IAM / Kubernetes RBAC / OPA / Vault に近い「MCP IAM / Control Plane」**。

### 2.2 ターゲットおよびユースケースの限定

- **定点観測(定型業務)と動的探索の切り分け:**
  - 条件が決まっている定点観測は、コストとエラー抑制のため「従来のバッチ処理」でデータを固め、最後の要約だけAIに投げる。
  - MCP GuardRailは、要因分析や突発的な調査などの**「動的な探索業務」**をターゲットとする。
- **提供形態:** 汎用YAML管理はAPIごとの特性(IAM、概念)が異なり二重管理で破綻するため、**「BigQuery安心パック」のように特定の主要APIに予め最適化されたプリセット設定済みのパッケージ/コンサル**として提供する。
  - **(v2追記)** この方針は §4.6 のIAM委譲マトリクス(コネクタごとに防御の最終ラインが異なる事実)によって技術的にも裏付けられる。

### 2.3 MVPスコープ(決定)

**BigQuery単体に全振りする。**

- 理由: コスト暴走は金額で示せる最も明確なペインであり、readonly強制・`maximumBytesBilled`・累積クォータ・監査ログだけで「導入前後のBigQuery請求額」という最強の実績数値が取れる。
- Slack/Notion等の情報漏洩系は別ペルソナ(情シス)の話であり、フェーズ1の実績作りには混ぜない。ポリシーエンジンのインターフェースだけコネクタ非依存に設計し、コネクタ追加はフェーズ1後半〜フェーズ2で行う。

### 2.4 成功指標(KPI)(v2新設)

フェーズ1の「実績」として対外発信に使える数値を最初から計測する。

1. **BigQueryスキャンコスト削減額/率**(導入前後比較)
2. **ブロックした危険リクエスト数**(破壊的SQL、期間無指定、閾値超過クエリの各件数)
3. **監査ログのカバレッジ**(AI経由の全クエリのうちログに記録された割合 = 100%であること)
4. 利用ユーザー数・チーム数(社内展開の広がり)

## 3. システムアーキテクチャ要件

### 3.1 3層構造(疎結合の原則)

システムは「フロント(クライアント)」「コアエンジン(GuardRail)」「コネクタ(API接続)」を完全に疎結合にする。

1. **フロントエンド層:** ユーザーが慣れたツール(Cursor, Claude Code, ChatGPT, 自社アプリ等)をそのまま利用。フロントを乗り換えても設定・ポリシーは不変。
2. **MCP GuardRail(中央集権Control Plane):** 認証、ルーティング、ポリシー検証、監査、クォータを一元管理する単一ポイント。
3. **コネクタ層:** 外部API(MVPではBigQueryのみ。将来: GA4, Slack, Notion, Backlog等)との純粋な接続(リクエスト送信)のみを担当。標準のOSS MCPサーバーは信用せず使わない(自社実装)。

内部コンポーネント分離(他AI案の採用):

```
Client (Claude Code / Cursor / ...)
        │  Streamable HTTP + OAuth
        ▼
┌─────────────────────────────┐
│  MCP GuardRail Control Plane │
│  ├─ Auth (IdP OAuth / セッション)  │
│  ├─ Policy Engine (YAML評価)      │
│  ├─ Quota Service (累積・状態管理)  │
│  ├─ Audit Service (改ざん困難ログ)  │
│  └─ Router                        │
└─────────────────────────────┘
        │
        ▼
  Connector層 (MVP: bigquery-connector)
        │
        ▼
  BigQuery API (最終防御: GCP IAM + RLS)
```

- 新コネクタ(Backlog, Jira等)追加時は**コネクタを書くだけ**で、認証・認可・監査・ポリシー適用・クォータは共通基盤が提供する。

### 3.2 デプロイ形態(決定)

- **社内共有サーバー(単一エンドポイント)+ Streamable HTTP。**
- stdio(各ユーザーのローカル起動)は不採用。理由: ポリシーYAMLと資格情報が各自のマシンに配られる形になり、中央集権Control Planeという核心価値(改ざん防止・一元監査・即時ポリシー反映)が成立しないため。
- MCPクライアント側の設定は「GuardRailのURL + OAuthログイン」のみ。クライアント乗り換え時も再設定不要。

### 3.3 認証・ユーザー識別(決定)

- **IdP(Google Workspace / Entra ID等)とのOAuth連携**(MCP仕様の認可フローに準拠)。
- **役職ロールのソースはIdPのグループ**とする。ポリシーYAMLに個人名を書かない(二重管理の禁止)。
- すべてのツール呼び出しは「認証済みユーザーのアイデンティティ」に紐づく。匿名呼び出しは存在しない。

### 3.4 状態管理(v2新設)

累積クォータ(§4.1)と監査ログ(§4.5)の要件により、GuardRailは**ステートレスなプロキシではなくDBを持つサービス**である。

- 保持する状態: ユーザー/日単位の累積スキャン量・呼び出し回数、監査ログ、セッション。
- ポリシー定義自体はDBに持たず、Git管理されたYAMLを正とする(§4.7)。

## 4. 機能要件(Level 3: 引数・業務ルールレベルの統制)

既存Gateway(Microsoft, Envoy, Docker等)のLevel 1(集約・認証)〜Level 2(ツール単位RBAC)を超え、市場の空白地帯であるLevel 3を実装する。

### 4.1 コストおよびデータ量のガードレール機能

| # | 要件 | 内容 |
|---|---|---|
| C-1 | 引数の必須化 | 期間指定(開始日・終了日)がないリクエストをバリデーションエラーとする。ツールのdocstringに「指定がない場合はユーザーに確認せよ」と自動挿入 |
| C-2 | プログラム側リミット(クランプ) | AIが膨大な件数を指定しても上限を上書き(例: `safe_limit = min(limit, 500)`) |
| C-3 | 期間最大幅の制限 | 一度に取得できる期間を「最大31日分」等に制限。超過時はAIに再考させる構造化エラーを返却 |
| C-4 | **ハードコスト上限(v2変更)** | BigQueryジョブに **`maximumBytesBilled`** を必ず付与し、閾値(例: 10GB)超過クエリを**課金前にBigQuery側で失敗**させる。これが統制の本体 |
| C-5 | ドライラン(v2で役割変更) | ドライランAPIで事前見積もりし、超過見込み時は「スキャン量◯GB、上限◯GB。期間を絞れ」等の**親切なエラーをAIに返すUX機能**。C-4のハード上限と二段構え(ドライランのみに依存しない — 見積もりと実行の間のズレ対策) |
| C-6 | **累積クォータ(v2新設)** | ユーザー/日単位の累積スキャン量(例: 100GB/日/人)・呼び出し回数の上限。リトライループによるコスト暴走を防止。超過時はその日のクエリ実行をdeny |

### 4.2 SQL統制(破壊的操作の禁止)(v2で方式変更)

- **旧方式(v1)**: `DROP`, `DELETE`, `ALTER`, `TRUNCATE` 等のキーワード検閲 → **不採用**。コメント挟み込み(`/**/`)・`EXECUTE IMMEDIATE`(動的SQL)・マルチステートメントで突破可能、かつ `delete_flag` 等のカラム名で誤検知するため。
- **新方式(S-1)**: SQLパーサー(sqlglot等)でASTを取得し、**「ステートメント種別が許可リスト(SELECTのみ)に一致すること」をホワイトリスト方式で検証**する。パース不能なSQLはdeny(fail-closed)。
- **S-2(多層防御)**: コネクタがBigQueryに対して使う資格情報自体を読み取り専用(`roles/bigquery.dataViewer` + `roles/bigquery.jobUser`。`dataEditor`は付与しない)に制限する。**GuardRailのパーサーを突破されてもIAMで破壊操作が物理的に不可能**な構成を必須要件とする。
- **S-3**: `EXECUTE IMMEDIATE`・マルチステートメントスクリプト・DDL/DMLはAST種別レベルで一律deny。

### 4.3 ユーザー・役割ベースのアクセス制御(ポリシーエンジン)

- **P-1**: ツール単位のallow/deny(Level 2相当。IdPグループ→ロールのマッピングに基づく)。
- **P-2**: 引数レベルのallow/deny(Level 3)。MVPではBigQuery: 参照可能な `datasets` の制限、`max_rows`、`readonly` 等。

```yaml
# policy/bigquery.yaml(イメージ)
bigquery:
  execute_sql:
    allow_roles: [analyst, engineer]
    statement_allowlist: [SELECT]
    datasets:
      allow: [analytics, marketing_public]
    max_rows: 1000
    max_bytes_billed: 10GB
    max_date_range_days: 31
    daily_quota_per_user: 100GB
```

- **P-3(将来コネクタの例、フェーズ1対象外)**: Slackの `channel="#finance"`/`#executive` への投稿・取得拒否、Notionの `engineering/*` 配下以外へのアクセス拒否、GitHubの `merge_pr` レビュアー数必須化、BacklogのIssue更新のみ許可等。
- **P-4**: deny時は `403 Forbidden` 相当の構造化エラーを返し、**AIが自律修正できる理由文**(例: 「このデータセットへの権限がない。analytics配下を使え」)を含める。
- **実装方式**: OPA/Regoは採用しない(学習コストが高く「コネクタ特化プリセット」思想と不整合)。**コネクタごとにスキーマ定義された自作YAML評価器**(スキーマバリデーション付き)とする。

### 4.4 コンテキスト・パススルー(トークンバケツリレー)

- 複雑なデータ権限はGateway側で二重管理せず、**ユーザー個人のOAuth資格をそのまま裏のAPIに引き継ぎ、データソース側のIAMで最終防御**する。
- **(v2追記)** BigQueryはユーザー資格委譲が可能なためこの方式が成立する。ただし全コネクタ共通の要件にはできない(§4.6マトリクス参照)。

### 4.5 監査ログ(v2新設 — 必須要件)

> v1では機能要件から漏れていたが、B2Bの購買理由(情シスの稟議)はブロック機能よりも監査要件であることが多い。Level 3を名乗る以上、引数レベルの監査ログは必須。

- **A-1**: 全ツール呼び出しについて「**誰が(IdPアイデンティティ)・いつ・どのクライアントから・どのツールを・どの引数(SQL全文含む)で呼び・許可/拒否/エラーのどれになったか・スキャン量/行数はいくつか**」を記録する。
- **A-2**: ログは追記専用(append-only)とし、利用者・管理者による改ざんが困難な保存先に置く。
- **A-3**: deny事象は理由(違反したポリシー項目)を含めて記録する。KPI(§2.4)の集計ソースを兼ねる。

### 4.6 コネクタ層要件とIAM委譲マトリクス(v2新設)

コネクタは「APIとの接続」だけを担当し、OAuth・Retry・(必要に応じ)Masking を実装する。ポリシー・監査・クォータは共通基盤に委ねる。

コネクタごとに「データソース側IAMに最終防御を委譲できるか」が異なるため、コネクタ追加時は必ず本マトリクスを更新する:

| コネクタ | ユーザー資格委譲 | 最終防御ライン | 備考 |
|---|---|---|---|
| BigQuery (MVP) | ◯ OAuthユーザー資格で実行可 | GCP IAM + 行レベルセキュリティ/認可済みビュー | GuardRailは第1防御、IAMが最終防御 |
| Slack | △ bot token運用が主流 | **GuardRailのポリシー** | ユーザー単位委譲は構成が限られる |
| Notion | ✕ ワークスペース単位トークンのみ | **GuardRailのポリシー** | GuardRailが唯一の防御線 |
| GA4 / Backlog 等 | 追加時に調査 | 追加時に決定 | |

### 4.7 動的コンテキスト挿入(旧: プロンプト・ファイアウォール)(v2で分離)

v1の「役職に応じて `WHERE date >= '2026-01-01'` を必ず含めよという指示を挿入する」機能は、**AIが指示を無視すれば突破されるため統制(セキュリティ機能)としては扱わない**。以下の2要件に分離する:

- **F-1(UX機能)**: 役職に応じた制約説明をツールdescriptionに動的挿入し、AIの無駄なdeny→リトライ往復を削減する。
- **F-2(統制機能)**: 行・期間レベルのアクセス制限が必要な場合、(a) ASTレベルでのWHERE句強制注入(クエリ書き換え)、または (b) BigQueryの行レベルセキュリティ/認可済みビューへの委譲、のいずれかで**プログラム的に強制**する。MVPでは (b) を推奨(データソース側で守る方が確実)。

### 4.8 ポリシー管理(Policy as Code)

- ポリシーはYAMLファイルで一元定義し、**Git管理(Pull Request、レビュー、履歴、差分、ロールバック)**する。
- GuardRailサーバーはGitリポジトリのYAMLを正として読み込む(マージ→全ユーザーに即時反映)。
- YAMLはコネクタごとのスキーマでCI時にバリデーションする(壊れたポリシーのデプロイ防止)。ポリシー読み込み失敗時はfail-closed(全deny)。

## 5. 非機能要件(v2新設)

| 項目 | 要件 |
|---|---|
| フェイルセーフ | ポリシー評価不能・パース不能・ポリシー読み込み失敗時は**deny(fail-closed)** |
| レイテンシ | ポリシー評価+ドライランのオーバーヘッドは対話的利用を損なわない範囲(目安: ポリシー評価 <100ms) |
| 可用性 | 社内単一障害点になるため、フェーズ1でも再起動容易性・ヘルスチェックを備える(HA構成はフェーズ2以降) |
| シークレット管理 | コネクタ用資格情報は環境変数直書きではなくSecret Manager等で管理。クライアント側には一切配布しない |
| 監査ログ保全 | append-only、最低保持期間を定める(社内規程に合わせる) |

## 6. セキュリティ設計原則(v2新設)

1. **多層防御:** GuardRailのポリシー(第1層)を突破されても、データソース側IAM(第2層)で致命傷にならない構成を常に維持する。GuardRail単独に依存しない。
2. **Fail-closed:** 判断できないものは通さない。
3. **ホワイトリスト優先:** 「危険なものを列挙してdeny」ではなく「安全なものを列挙してallow」。
4. **プロンプトはUX、コードが統制:** AIへの指示文はガードレールではない。強制はすべてプログラム側で行う。
5. **最小権限:** コネクタの資格情報は目的に必要な最小ロールのみ。

## 7. 開発・生存ロードマップ(個人事業主としての2段階/ハイブリッド戦略)

現在の年齢、家族、ライフプラン、および「実績のない個人からセキュリティ製品は買わない」というB2B市場の特性を考慮し、リスク最小・リターン最大のステップを踏む。

### フェーズ1: 勤務先を「実験場・実績作りの場」として活用(社内プロジェクト化)

- 勤務先のBigQueryコスト削減、Cursor導入時の情報漏洩対策という名目で、社内インフラとして『MCP GuardRail』のプロトタイプ(= 本書のMVP: BigQuery安心パック)を開発・導入。
- 「実社内環境で稼働し、コストを◯%削減、セキュリティリスクをゼロにした」という本物のエンタープライズ事例(実績)を会社の金で獲得(計測項目は §2.4 KPI)。
- 得られた知見を社外ブログ等で発信し、「MCPガバナンスの専門家」としての個人の認知を作る。

#### フェーズ1の社内での立ち回り(2026-07-02追記)

本テーマは現状「自分の業務管轄外」であり、かつ**別の担当者が近い取り組みを進めている**。この状況で自分の業務として引き受けるには、提案の順番と開示範囲を誤らないこと。以下の順で進める。

1. **既存担当者との交通整理(上司に言う前にやる):** 縄張り争い(turf war)に見せないため、まず相手が実際に何をやっているかを把握し、衝突を解消する。上司が評価するのは「奪いに来る人」ではなく「衝突を整理して前に進める人」。相手の取り組みは次の3パターンで対応を変える。
   - *実は同じもの* → 奪わず**合流・協力**する。
   - *隣接だが別物* → 「相手はA、自分はB」と**差別化を用意してから**提案する。
   - *頓挫/手が回っていない* → ここで初めて「引き取る」提案が成立する。
2. **社内便益に絞って上司に提案:** 語るのは会社の具体的ペイン(BigQueryコストが読めない/Cursor導入時の情報漏洩)と、それを安全に制御する仕組みを試作したい、という点のみ。「既存担当者とは重ならないよう整理済み」と添える。
3. **事業化構想は口にしない(開示範囲の統制):** 「将来これを個人事業/SaaSにする」という意図は上司・社内に明かさない。理由は下記「IPの割り切り」。

#### IPの割り切り(前提として受け入れること)

- 会社の業務として作る以上、**成果物のIPは基本的に会社に帰属**する(職務発明等)。これは「信用・実績・会社リソース・本物のエンタープライズ事例」と引き換えに受け入れる前提。
- したがって**フェーズ2のクリーンルーム・リビルドは「任意」ではなく「必須」**。会社で書いたコードは会社のもの、と割り切り、個人版は設計思想・一般知識のみを持ち出して公開情報からゼロから組み直す(§7フェーズ2、および付録B参照)。
- 事業化意図を社内で明かすと、会社がIPを囲い込みにかかりフェーズ3の芽を潰す/「片足を外に出している人」と見られ社内信用を損なう、という二重のリスクがある。だから③で開示しない。

### フェーズ2: コアロジックの抽象化と個人でのリビルド(完全合法化)

- 業務時間中に書いたコードの権利は会社に帰属するため、そこで得た「知見・ノウハウ・設計思想」をベースに、休日に個人用のコードとしてイチから書き直す(リビルド)。
- まずは機能を絞ったミニマムなスクリプトを、コンセプトPoCとしてGitHubにOSS等で公開。
- **(v2追記)** 就業規則の職務発明・競業関連条項の事前確認をフェーズ1開始前のTODOとする。

### フェーズ3: 商品化・独立

- フェーズ1の実績を引っ提げ、「実績のある専門家」として、AIを導入したいがコストと漏洩が怖くて踏み切れない中小企業をターゲットに「MCP GuardRail導入支援パッケージ(受託コンサル・SaaS)」を販売する。

## 付録A: v1からの変更点と理由

| 変更 | 理由 |
|---|---|
| キーワード検閲(`DROP`等のdenyリスト)→ ASTホワイトリスト方式(§4.2) | コメント挟み込み・動的SQL・マルチステートメントで突破可能、カラム名での誤検知もあるため。denyリストは統制として成立しない |
| ドライラン単独 → ドライラン(UX)+ `maximumBytesBilled`(統制)(§4.1 C-4/C-5) | ドライラン→実行の2段階には見積もりと実行のズレがある。ハード上限はBigQuery側でアトミックに効かせる |
| プロンプト・ファイアウォール → UX機能と統制機能に分離(§4.7) | AIへの指示挿入は無視されれば突破される。「お願い」をセキュリティ機能として売ると事故る |
| 監査ログを必須機能要件として新設(§4.5) | v1では機能要件から漏れていた。B2Bの購買理由の中核であり、Level 3の根拠となる機能 |
| 累積クォータの新設(§4.1 C-6) | 1リクエスト単位のクランプではリトライループのコスト暴走を防げない |
| IAM委譲マトリクスの新設(§4.6) | 「トークンバケツリレー」はコネクタごとに実現可能性が異なる。プリセット提供型ビジネスの技術的裏付けにもなる |
| デプロイ形態・認証方式・MVPスコープの決定(§0, §2.3, §3.2, §3.3) | 2026-07-02の意思決定を反映 |
| 非機能要件・セキュリティ設計原則の新設(§5, §6) | fail-closed等、設計の判断基準を明文化 |
